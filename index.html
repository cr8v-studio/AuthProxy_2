You're Assembling Auth Infrastructure From 5 Different Services

Building a modern web application? You need:

  • An identity provider (Auth0, Keycloak, Firebase)
  • A reverse proxy (nginx, Traefik)
  • A file storage service (S3, MinIO)
  • A notification system (Firebase, OneSignal)
  • An admin panel (custom build or Retool)

That's 5 vendors, 5 failure points, 5 learning curves, and weeks of integration work. AuthProxy replaces all five. Single ASP.NET process. No external dependencies. Production-ready from day one.

One Process. Five Capabilities.

Internet → AuthProxy (port 443)
              ├── Authentication (11 methods)
              ├── Reverse Proxy (dynamic routing)
              ├── File Service (upload/download/images)
              ├── Notifications (SSE, Webhooks, Push)
              └── Admin Panel (13 pages)
                     ↓
              Backend Services (your Core, TrexWallet, Chat, CRM)

AuthProxy sits at the edge of your application. Every request passes through it. Authentication is verified, headers are injected (X-Crm, X-Scopes, X-Project), and requests are forwarded to your backend services. Your services never see unauthenticated traffic.

Every Auth Method Your Users Need. Already Built.

Cryptographic (Strongest)

  • PassKey: Ed25519 challenge-response, SHA256(phone+password+domain)
  • FIDO2/WebAuthn: Hardware keys with sign-count anti-cloning
  • UserKey: Imported Ed25519 hex private key

One-Time Passwords

  • Phone OTP: SMS/Messenger delivery, optional account creation
  • Email OTP: Email with MX validation + Magic Link option

Social / OAuth

  • Google, Telegram, Discord, Facebook, GitHub, VK, Apple

AI / Machine

  • MCP Basic: Public key headers, read-only
  • MCP Verified: Ed25519 timestamp signing
  • MCP App: Delegated third-party access

Key Technical Details

  • Challenge-Response Protocol: No passwords stored, 2-minute replay-safe TTL
  • In-Memory Sessions: ConcurrentDictionary O(1), instant revocation
  • Domain Salting: stolen credentials don't work across domains
  • Timing Protection: random 50-150ms delay on all auth attempts

Dynamic Routing. Zero-Downtime Config Changes.

Routes are stored in the route_map database table and reloaded every 100 seconds.

  • Three route types: SPA/PWA, API proxy, internal AuthProxy APIs
  • Priority-based matching and conflict handling
  • Security flags: ScopeCheck, WhiteListIPCheck, GeoIP, NoRateLimit, AppAuthAndCORS
  • Injected headers: X-Crm, X-Scopes, X-Project, X-KeyType, X-Country
  • Per-session rate limiting: 20 request buffer, 1/sec replenish
  • /private/* APIs restricted to internal networks
